# 后端开发计划：AI 视觉对话助手 MVP

## 背景

当前 `packages/backend/` 只有一个最小 Express 5 骨架（健康检查端点 + HTTP 服务器）。需要实现 MVP 开发文档中定义的 8 个后端模块，打通"用户语音 + 摄像头画面 → 豆包 ASR → LangGraph → GLM-5V-Turbo → 豆包 TTS → 前端播放"的完整闭环。

---

## 架构决策

| 决策 | 选择 | 理由 |
|---|---|---|
| LangGraph | 使用官方 `@langchain/langgraph` 的 `StateGraph` + `Annotation.Root()` | 规格文档明确使用 LangGraph；`StateGraph` API 简洁，线性链代码量与自定义相当；内置流式输出、状态管理、未来可扩展条件边和子图 |
| 共享类型包 | MVP 暂不创建 `packages/shared/` | 仅有 ~15 个 WebSocket 事件类型，前后端各定义一份即可，后续可提取 |
| WebSocket 路径 | 复用 HTTP 服务器的 `/ws` 路径 | 无需额外端口，Vite 代理已配置好 |
| Session ID | `crypto.randomUUID()` | Node 18+ 内置，无需外部依赖 |
| 环境变量 | zod 解析 + 启动时校验 | fail-fast，类型安全 |
| 日志 | 自定义轻量 console wrapper | 规格说明"只输出基础日志"，Pino/Winston 过度 |
| 测试框架 | Vitest | ESM 原生、与 tsx 兼容、快速 |
| 模块系统 | ESM + NodeNext（已配置） | 保持现有配置不变 |

## 开发阶段

| 文件 | 说明 |
|---|---|
| `src/config/env.ts` | 用 zod 解析和校验 `PORT`、`DOUBAO_ASR_*`、`DOUBAO_TTS_*`、`GLM_*` |
| `src/lib/logger.ts` | `logger.info/warn/error/debug(module, msg, meta?)`，ISO 时间戳 + 模块前缀 |
| `src/lib/errors.ts` | `AppError` 基类（statusCode + code），子类：`AsrError`、`TtsError`、`GlmError`、`SessionError` |
| `src/types/events.ts` | WebSocket 协议类型：6 个 ClientEvent + 9 个 ServerEvent（discriminated union） |
| `src/types/session.ts` | `SessionState` 枚举、`Session` 接口 |
| `src/types/workflow.ts` | `WorkflowState`、节点输入/输出类型 |

**依赖安装**：`pnpm --filter backend add zod`

### Phase 1：会话管理 + 状态机 + 帧存储

**目标**：实现半双工状态机和内存会话管理，可完全通过单元测试验证。

| 文件 | 说明 |
|---|---|
| `src/session/sessionStore.ts` | `Map<string, Session>`：create/get/delete/getAll |
| `src/session/sessionStateMachine.ts` | 纯函数 `transition(state, event) → newState`，验证 6 状态 × 6 事件 的合法转移 |
| `src/session/sessionManager.ts` | 组合 store + 状态机 + latestFrame 存取，对外暴露业务方法 |

**关键逻辑**：
- `canAcceptAudio(id)` → 仅 `listening` 状态返回 true
- 非法状态转移抛出 `SessionError`

### Phase 2：WebSocket 网关 + 连接管理

**目标**：建立 WebSocket 连接管理，解析消息并路由到 handler。

| 文件 | 说明 |
|---|---|
| `src/gateway/connectionManager.ts` | 单例：connections Map、add/remove/get、`sendToSession(id, event)` |
| `src/gateway/wsGateway.ts` | 创建 WSS（path: `/ws`）、消息解析、handler map 路由分发 |
| `src/routes/health.ts` | 从 index.ts 提取 |
| `src/index.ts` | **重构**：集成 env 配置、健康路由、WS 网关、错误中间件 |

**6 个 client event handler（初期桩实现）**：
- `session.start` → sessionManager.createSession() → 回复 session.ready
- `audio.chunk` → 检查 canAcceptAudio → 转发 ASR（Phase 3 接入）
- `frame.update` → sessionManager.setLatestFrame()
- `turn.end` → transition transcribing
- `playback.done` → transition listening
- `session.stop` → transition closed → cleanup

### Phase 3：豆包流式 ASR 服务

**目标**：实现豆包流式语音识别客户端。

| 文件 | 说明 |
|---|---|
| `src/services/doubaoAsr.ts` | DoubaoAsrService 类 |

**接口**：
- `startRecognition(sessionId)` — 建立与火山引擎 ASR 的 WebSocket（二进制协议）
- `sendAudioChunk(sessionId, chunk: Buffer)` — 转发音频数据
- `stopRecognition(sessionId): Promise<string>` — 发送结束信号，等待 final 结果
- 回调：`onPartial(sessionId, text)`、`onFinal(sessionId, text)` → 通过 wsGateway 推送给前端

**集成点**：Phase 2 的 `audio.chunk` handler → doubaoAsr.sendAudioChunk；`turn.end` handler → doubaoAsr.stopRecognition

### Phase 6：LangGraph 工作流

**目标**：使用官方 `@langchain/langgraph` 实现 5 节点线性工作流，编排一次 AI 回复的完整流程。

**核心 API**：`StateGraph` + `Annotation.Root()` 定义共享状态，`.addNode()` + `.addEdge()` 构建线性链，`.compile()` 生成可执行图。

| 文件 | 说明 |
|---|---|
| `src/workflow/state.ts` | `Annotation.Root()` 定义 WorkflowState（各字段 + reducer），导出 `State` 对象和类型 |
| `src/workflow/nodes.ts` | 5 个节点函数，签名 `(state: StateType) => Promise<Partial<StateType>>`，通过闭包注入服务依赖 |
| `src/workflow/graph.ts` | 构建 `new StateGraph(State)` → addNode × 5 → addEdge START→node1→...→node5→END → `.compile()` |

**关键代码结构**：
```typescript
// state.ts
import { Annotation } from "@langchain/langgraph";

export const WorkflowState = Annotation.Root({
  sessionId: Annotation<string>(),
  userText: Annotation<string>(),
  latestFrameBase64: Annotation<string | undefined>(),
  assistantText: Annotation<string | undefined>(),
  assistantAudio: Annotation<Buffer | undefined>(),
  error: Annotation<string | undefined>(),
  asrCalls: Annotation<number>({ default: () => 0, reducer: (_, next) => next }),
  glmCalls: Annotation<number>({ default: () => 0, reducer: (_, next) => next }),
  ttsCalls: Annotation<number>({ default: () => 0, reducer: (_, next) => next }),
});

// graph.ts
import { StateGraph, START, END } from "@langchain/langgraph";

export function createMvpGraph(deps: WorkflowDeps) {
  return new StateGraph(WorkflowState)
    .addNode("receiveTurn", receiveTurnNode)
    .addNode("loadLatestFrame", loadLatestFrameNode(deps))
    .addNode("callGlm5vTurbo", callGlm5vTurboNode(deps))
    .addNode("synthesizeWithDoubaoTts", synthesizeWithDoubaoTtsNode(deps))
    .addNode("recordUsage", recordUsageNode(deps))
    .addEdge(START, "receiveTurn")
    .addEdge("receiveTurn", "loadLatestFrame")
    .addEdge("loadLatestFrame", "callGlm5vTurbo")
    .addEdge("callGlm5vTurbo", "synthesizeWithDoubaoTts")
    .addEdge("synthesizeWithDoubaoTts", "recordUsage")
    .addEdge("recordUsage", END)
    .compile();
}
```

**5 个节点**：
1. `receiveTurn` — 标准化输入数据（sessionId、userText、latestFrameBase64）
2. `loadLatestFrame(deps)` — 从 sessionManager 读取最新帧，注入到 state
3. `callGlm5vTurbo(deps)` — 调用 GLM 服务 → 设置 assistantText
4. `synthesizeWithDoubaoTts(deps)` — 调用 TTS 服务 → 设置 assistantAudio
5. `recordUsage(deps)` — 更新 asrCalls/glmCalls/ttsCalls 计数器

**流式输出**：LangGraph 内置 `streamMode` 支持 `"values"` / `"updates"` 模式，可在每个节点完成后向前端推送进度事件（如 `assistant.thinking` → `assistant.text` → `assistant.audio`）。

**集成点**：ASR final 完成后 → sessionManager.recordAsrFinal() → 触发 thinking 状态 → `graph.invoke(initialState)` 或 `graph.stream()` 运行 workflow → 推送 assistant.text/audio/done/usage.update

## 测试策略

### 测试框架

在 `packages/backend/package.json` 添加 **Vitest**（ESM 原生、与 tsx 兼容）：
```
pnpm --filter backend add -D vitest @vitest/coverage-v8
```

添加脚本：`"test": "vitest run"`、`"test:watch": "vitest"`

### 测试优先级

| 优先级 | 测试对象 | 类型 | 理由 |
|---|---|---|---|
| 🔴 最高 | sessionStateMachine | 单元 | 纯逻辑，6×6 状态转移矩阵，核心正确性 |
| 🔴 最高 | workflow/graph | 单元 | 用 mock deps 构建 graph，验证 5 节点顺序执行和状态累积 |
| 🟡 高 | sessionManager | 单元 | 组合逻辑，验证状态流转 + 帧存储 |
| 🟡 高 | usageRecorder | 单元 | 累加逻辑 |
| 🟢 中 | wsGateway 协议 | 集成 | 用 ws 客户端连接，验证消息往返 |
| 🟢 中 | workflow/graph | 集成 | 用 mock 服务验证完整 5 节点流程 |
| ⚪ 外服务 | ASR/TTS/GLM | 手动 | 需真实 API Key，开发时手工验证 |

## 依赖变更

### 新增 production 依赖
- `zod` — 环境变量校验（需显式声明为直接依赖）
- `@langchain/langgraph` — 官方 LangGraph.js，使用 `StateGraph` + `Annotation.Root()` 构建 5 节点工作流

### 新增 dev 依赖
- `vitest` — 测试运行器
- `@vitest/coverage-v8` — 代码覆盖率

### 不需要安装
- `@langchain/core` — LangGraph 已内置所需的核心类型，无需单独安装
- `@langchain/openai` / langchain LLM 集成 — 使用自定义 HTTP 客户端调用 GLM 和豆包
- `pg` / ORM — MVP 不启用数据库
- `pino` / `winston` — 规格说明只要基础日志
- `socket.io` — 使用已有的 `ws` 原始 WebSocket

## 验证方式

1. **Phase 0-1**：`pnpm --filter backend test` 通过（状态机 + store 单元测试）
2. **Phase 2**：`pnpm dev:backend` 启动，用浏览器 WebSocket 或 `websocat` 连接 `ws://localhost:3001/ws`，发送 `{"type":"session.start"}` 验证收到 `session.ready`
3. **Phase 3-5**：单独测试每个服务，用真实 API Key 调用验证（手动）
4. **Phase 6-7**：集成测试通过（workflow + usage）
5. **Phase 8**：全链路测试：前端连接 → 发送音频和图片 → 收到 AI 文本和语音 → 播放完成恢复录音
6. **前端联调**：`pnpm dev` 启动全栈，打开浏览器验证 18 条验收标准
